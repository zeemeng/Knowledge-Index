Hypertext Transfer Protocol (HTTP) is a application layer network protocol for transferring data between hypermedia information systems. Contrary to what the name would suggest, HTTP is not exclusive used of the transfer of textual data. In fact, most types of media may be transferred over HTTP. HTTP can be described as a request-response stateless protocol. The term “request-response” means that the mode of communication involves a message exchange pattern where a first network node behaving as a client sends a request for some resource to a second network node behaving as a server, which upon receipt of the request sends back a response to the first node. The term “stateless” means that “no session information is retained by the receiver/server. Relevant session data is sent to the receiver by the client in such a way that every packet of information transferred can be understood in isolation, without context information from previous packets in the session”. Each request-response cycle is essentially a Markov process.
The typical steps involved in the communication process using HTTP is as follows:
  1. Client opens a connection to the server. (Usually over TCP)
  2. Client sends an HTTP request for a resource to the server.
  3. Server sends back an appropriate response to the client.
  4. Either the client or the server closes the connection or the connection is kept alive for further communications.
Collectively, HTTP requests and responses are called “HTTP messages” and the structure a HTTP message is as follows :
  1. A “start-line”, which can be either a “request-line” or a “status-line”, followed by a carriage return character (CR) and a line feed character (LF)
  2. A message header (each header field is followed by CRLF)
  3. An empty line (i.e. CRLF)
  4. An optional message body depending the nature of the message
The first component of an HTTP message is a start-line, which is differentiated into request-line for request messages and status-line for response messages. The structure of a request-line is as such: the first element is a HTTP method (i.e. a specific command, see below for details) followed by a space character and a URI that is the target of the request, then followed by space character and the protocol version and finally ending with a CRLF. In the case of status-line, the structure is as such: the first element is the protocol version followed by a space character and the status code (see below for details), then followed by a space and a textual description of the status code and finally ending with a CRLF.
The second component of an HTTP message is a message header. It is composed of a number of header fields which are name-value pairs. A header field is structured as a “name” string, followed by a colon character, followed by a value string, and each header field is ended with a CRLF. The number of header fields present may vary according to each message being sent. However, most header fields are optional. While in the HTTP/1.1 protocol, for request messages, the Host header field is the only one that is mandatory in order to distinguish “between various DNS names sharing a single IP address”, accommodating “name-based virtual hosting”.
The message header is immediately followed by an empty line (i.e. a CRLF). Then, the empty line may be followed by an optional message body which presence depends on whether the message is a request or a response, as well as on the HTTP method used in the request-response cycle.

Each HTTP request message is associated with a particular HTTP method, which specifies the type of action to be carried out by the receiver of the message. As mentioned above, the method is specified as the first element of the request-line and is therefore also the first element of the message. It is a case sensitive string token and it's possible values as defined by technical standards published by the IETF are: GET , HEAD , POST , PUT , DELETE , CONNECT , OPTIONS , TRACE , PATCH . Below is a description of a some of the most frequently used ones: GET , POST , PUT , DELETE . GET retrieves data from the receiver/server. POST submits data to the receiver/server. The data to be submitted shall be given in the request message body. PUT updates data already on the receiver/server. The data to be used as update shall be given in the request message body. DELETE deletes data from the receiver/server. When the receiver/server network node receives any request message, it can either choose to accept and process the request or it can choose to refuse it for some given reason. In either case, the server shall send back a response message to the client with a status code describing the state of the request. As mentioned above, the status code is specified as the second element on the status-line of the response message. HTTP status codes are 3-digit tokens categorized into ranges to indicate different classes of response state. These categories of status codes are distinguished by the first digit and are as follows :
  • 1xx : Informational - Request received, continuing process
  • 2xx : Success - The action was successfully received, understood, and accepted
  • 3xx : Redirection - Further action must be taken in order to complete the request
  • 4xx : Client Error - The request contains bad syntax or cannot be fulfilled
  • 5xx : Server Error - The server failed to fulfill an apparently valid request
In the case of a successful response (i.e. with a 2xx status code) to a GET request, the requested resource is given in the response message body.
