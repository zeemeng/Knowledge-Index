CSRF threats may also be mitigated using CSRF tokens which are unique tokens generated by a web server when a client requests a particular resource from the server. After a CSRF token is generated, it is included in the response that the server sends back to the client. If the application protocol used in such a transaction is HTTP , the CSRF token may be included in a response header field (with the exception of "Set-Cookie") or in the response body (e.g. embedded as the value of a hidden field within an HTML form element). Once the client receives the response containing the token, the token may be extracted and stored client-side at some location such as in a variable of the JS environment, in "session storage", in the HTML document etc. However, crucially, the CSRF token should not be stored as a cookie, as it would defeat the pupose of such a CSRF mitigation system. Subsequently, when a legitimate request to a protected resource located on server is issued from the client-side application, the latter would be responsible to attach the given CSRF token as part of the request. As such, when the server receives the request, it is able to inspect and validate the included CSRF token, thus distinguishing requests originating from a legitimate client application from those that do not.
